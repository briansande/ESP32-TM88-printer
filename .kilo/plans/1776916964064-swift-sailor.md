# Plan: Gamma Correction for Image Pipeline

## Context

The image pipeline currently works as follows:
- **Client-side (JS in `web_server.cpp` HTML):** Image → greyscale → dither (Floyd-Steinberg or threshold) → bitpack into 1-bit bytes → base64 → POST to ESP32
- **Server-side (ESP32):** Decode base64 → `PrinterDriver::printImage()` → printer

There is no C++ image processing on the ESP32 yet. The user wants an LUT-based gamma correction function in C++ that operates on greyscale `uint8_t` buffers before dithering.

## Changes

### 1. Create `include/image_pipeline.hpp`

Declare a free function:
```cpp
void gammaCorrect(uint8_t* pixels, size_t len, float gamma = 1.8f);
```

Header-only is unnecessary since we have a `.cpp` pair. Keep it minimal — no class, just a utility function.

### 2. Create `src/image_pipeline.cpp`

Implement LUT-based gamma correction:
```cpp
#include "image_pipeline.hpp"
#include <cmath>

void gammaCorrect(uint8_t* pixels, size_t len, float gamma) {
    uint8_t lut[256];
    float invGamma = 1.0f / gamma;
    for (int i = 0; i < 256; i++) {
        lut[i] = (uint8_t)(round(pow(i / 255.0f, invGamma) * 255.0f));
    }
    for (size_t i = 0; i < len; i++) {
        pixels[i] = lut[pixels[i]];
    }
}
```

- `pow()` is called exactly 256 times (once per LUT entry), never per-pixel.
- The LUT is a stack-allocated `uint8_t[256]` — zero heap allocation.
- The correction formula: `out = round((in/255)^(1/gamma) * 255)` — this darkens midtones for gamma > 1.0, which compensates for thermal printer over-blooming.

### 3. Add gamma correction to the client-side JS pipeline

In the `processImage()` function in `web_server.cpp`, insert a gamma correction step **after** greyscale conversion and **before** dithering. This gives immediate benefit without requiring the ESP32 to do the processing.

Add a gamma input control to the HTML:
- A numeric input (default 1.8, range 0.1–5.0) in the "Image Print" section
- The `processImage()` function applies a JS LUT-based gamma correction on the `Float32Array` greyscale buffer before dithering

### 4. Wire C++ gamma correction into `handlePrintImage()` (optional future path)

The C++ `gammaCorrect()` function will be available in `image_pipeline.hpp` for when the user wants to move image processing server-side. No wiring in this change — the function is ready to include and call. A brief comment in the header will note the intended usage.

## Notes

- `pow()` from `<cmath>` is used only 256× at LUT build time — zero per-pixel cost.
- The LUT is 256 bytes on the stack — negligible ESP32 memory impact.
- Default gamma of 1.8f darkens midtones to compensate for thermal printer over-blooming.
- Gamma < 1.0 lightens midtones (inverse curve), gamma = 1.0 is identity.
